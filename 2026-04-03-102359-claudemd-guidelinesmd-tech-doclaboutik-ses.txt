# Exploration — Rapports bilan de billetterie                                   
                                                                                
## Objectif                                                                     
                                                                                
Explorer et concevoir un système de rapports "bilan" pour la billetterie.       
**Ne code rien.** Explore le codebase, comprends ce qui existe, et propose un   
design.                                                                         
                                                                                
Charge les skills /djc et /brainstorming avant de commencer.                    
                                                                                
## Contexte projet                                                              
                                                                                
TiBillet/Lespass est un groupware multi-tenant (django-tenants) pour la         
billetterie,                                                                    
les adhésions et le cashless NFC. La billetterie est dans `BaseBillet/`         
(TENANT_APP).                                                                   
L'admin est Django Unfold. Le frontend est HTMX + Bootstrap 5.                  
                                                                                
Le module LaBoutik (caisse/POS) a déjà des rapports comptables riches           
(sessions 12-21 : RapportComptableService, clôtures J/M/A, export FEC/CSV).     
La billetterie n'a **rien d'équivalent** — pas de rapport de synthèse.          
                                                                                
## Ce qu'on veut savoir dans un bilan de billetterie                            
                                                                                
Pour un événement (ou une période) :                                            
                                                                                
### Ventes                                                                      
- Nombre de billets vendus par tarif (plein, réduit, gratuit, invitation...)    
- Chiffre d'affaires TTC/HT par tarif                                           
- Ventilation par moyen de paiement (Stripe, espèces caisse, HelloAsso...)      
- Évolution des ventes dans le temps (courbe J-30 → J)                          
- Taux de remplissage (vendus / jauge max)                                      
                                                                                
### Remboursements / Annulations                                                
- Nombre et montant des remboursements (Stripe refund)                          
- Nombre d'annulations (sans remboursement)                                     
- Taux de remboursement                                                         
                                                                                
### Scan / Accès                                                                
- Nombre de billets scannés (entrées effectives)                                
- Taux de no-show (vendus - scannés)                                            
- Horaire des scans (courbe d'affluence)                                        
                                                                                
### Codes promo                                                                 
- Utilisation par code promo                                                    
- Montant de réduction accordé                                                  
                                                                                
### Par canal de vente                                                          
- Ventes en ligne (Stripe) vs ventes caisse (LaBoutik) vs HelloAsso             
- Si applicable : ventes par point de vente

  2. Proposition d'architecture

  Le service : BaseBillet/reports.py

  Même pattern que RapportComptableService : une classe initialisée avec un
  event (ou une période), des méthodes qui retournent des dicts. Calcul temps 
  réel à chaque affichage (pas de snapshot — les données d'un event terminé ne
  bougent plus).

  class RapportBilletterieService:
      def __init__(self, event):
          # Filtre de base : LigneArticle VALID + REFUNDED liées à cet event
          # via reservation__event=event
          self.lignes_valides = LigneArticle.objects.filter(
              reservation__event=event,
              status__in=[LigneArticle.VALID, LigneArticle.REFUNDED]
          ).select_related('pricesold__price',
  'pricesold__productsold__product', ...)

      def calculer_synthese()          # totaux globaux
      def calculer_ventes_par_tarif()  # GROUP BY price
      def calculer_par_moyen_paiement() # GROUP BY payment_method
      def calculer_par_canal()         # GROUP BY sale_origin
      def calculer_scans()             #
  Ticket.objects.filter(reservation__event=...)
      def calculer_codes_promo()       # GROUP BY promotional_code
      def calculer_remboursements()    # LigneArticle.status=REFUNDED +
  credit_note_for

  Où afficher — 3 options

  Option A (recommandée) : Carte dans l'admin Event, change_form_before_template
  - Cohérent avec le pattern existant (export fiscal dans ClotureCaisseAdmin)
  - La carte "Bilan de billetterie" s'affiche au-dessus du formulaire Event
  - Un bouton "Export PDF" et "Export CSV" dans la carte
  - HTMX inline pour charger à la demande (pas à chaque ouverture de l'admin)

  Option B : Page dédiée /admin/basebillet/event/{uuid}/bilan/
  - Via get_urls() dans EventAdmin + admin_site.admin_view()
  - Plus d'espace, meilleur pour un rapport long
  - Lien "Voir le bilan" dans les actions_row de la liste Event

  Option C : Section Unfold TableSection
  - Comme EventPricesSummaryTable existante
  - Limité : pas facile d'y mettre des exports PDF, plusieurs sections

  Recommandation : Option B (page dédiée) pour le rapport complet + lien dans la
   liste. Option A pour un aperçu rapide dans la fiche Event.

  Exports

  - PDF via WeasyPrint (même pattern que les clôtures laboutik)
  - CSV délimiteur ; (même pattern)
  - Action via get_urls() dans EventAdmin (pattern recommandé par les guidelines
   Unfold — évite les bugs de permission actions_row)

  4. Questions ouvertes pour le mainteneur

  Q1 — Timestamp de scan
  Le modèle Ticket n'a pas de scanned_at. La courbe d'affluence horaire est donc
   impossible. Est-ce qu'on ajoute ce champ (migration), ou on se passe de la
  courbe ?

  Q2 — Périmètre du rapport : événement seul ou aussi période ?
  RapportComptableService fonctionne sur une plage de dates. Pour la
  billetterie, le rapport par event semble plus naturel. Doit-on aussi pouvoir
  faire un rapport "toutes les ventes billetterie du mois de mars"
  (multi-events) ?

  Q3 — Où afficher dans l'admin ?
  - Option A : carte dans la fiche Event (change_form_before_template) → aperçu
  rapide
  - Option B : page dédiée /admin/event/{uuid}/bilan/ → rapport complet, plus
  d'espace

  Les deux ne sont pas exclusifs. Laquelle implémenter en premier ?

  Q4 — HelloAsso
  Il n'y a pas de SaleOrigin.HELLOASSO dans les choices. Les ventes HelloAsso
  passent-elles par EXTERNAL ou WEBHOOK ? Faut-il ajouter une valeur dédiée ?

  Q5 — Remboursements
  Les remboursements Stripe ne sont pas stockés en base (stripe.Refund.create()
  est appelé mais pas sauvegardé). On peut tracer via
  LigneArticle.status=REFUNDED et LigneArticle.credit_note_for. Est-ce
  suffisant, ou faut-il stocker l'ID du refund Stripe ?

  Q6 — Events hiérarchiques
  Event.parent (FK→self) permet des sous-events. Le bilan doit-il agréger
  automatiquement les sous-events dans le bilan du parent ?
