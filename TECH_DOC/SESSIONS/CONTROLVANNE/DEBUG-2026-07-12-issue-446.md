# DEBUG 2026-07-12 — Issue #446 : impossible de créer une carte de maintenance

Issue : https://github.com/TiBillet/Lespass/issues/446
Labels : `stress-test-V2`, `module:tireuse`
Signalée hors grille depuis la régie par **Starsky**.

## Symptôme rapporté

Sur `/admin/controlvanne/cartemaintenance/add/` :
- les deux Tag ID saisis (`611377E9`, `CAF48622`) ne sont **pas reconnus**, alors
  que les cartes venaient d'être créées ;
- la loupe ouvre « un popup pleine page de création de carte »
  (`/admin/QrcodeCashless/cartecashless/?_to_field=id&_popup=1`).

Attendu : une liste déroulante ou une complétion de saisie.

## Root cause

`controlvanne/admin.py` — `CarteMaintenanceAdmin` déclarait :

```python
raw_id_fields = ("carte",)
```

…alors que sa **propre docstring, deux lignes plus haut, annonçait
`autocomplete_fields`**. Le code ne faisait pas ce que le commentaire promettait.

`raw_id_fields` rend un `UnfoldForeignKeyRawIdWidget` : un champ texte qui attend
le **pk numérique** de la `CarteCashless` (`1`, `2`…). Un Tag ID (8 hexa) n'y est
évidemment jamais reconnu. Et la loupe de ce widget ouvre le changelist des
cartes en popup — ce que Starsky a pris pour un formulaire de création.

Reproduction en shell, avant fix :

```
WIDGET : UnfoldForeignKeyRawIdWidget
autocomplete_fields = ()
raw_id_fields = ('carte',)
```

## Second défaut, trouvé pendant l'investigation (non signalé)

`CarteMaintenanceAdmin` n'avait **aucun `formfield_for_foreignkey`**.

`CarteCashless` est en SHARED_APPS (schéma `public`) : **aucune isolation
automatique**. Le queryset du champ remontait donc les cartes de **tous les
lieux**. Or le queryset d'un champ est ce qui **valide** la valeur postée : un pk
forgé aurait rattaché la carte d'un autre lieu comme carte de maintenance.

Le test d'isolation écrit avant le fix le prouve :
`assert 10 not in {1, 2, 3, 4, 9, 10}` — la carte du lieu B était sélectionnable
depuis `lespass`.

`CartePrimaireAdmin` (`Administration/admin/laboutik.py:268`) documente très
précisément ce piège et le corrige. `CarteMaintenance` se présente comme le
« même patron que CartePrimaire » (docstring du modèle) : le patron a été copié
sans sa protection.

## Fix

| Fichier | Changement |
|---|---|
| `controlvanne/admin.py` | `raw_id_fields` → `autocomplete_fields = ("carte",)` ; ajout de `formfield_for_foreignkey` filtrant sur `detail__origine=connection.tenant` ; imports `connection` + `CarteCashless` |
| `tests/pytest/test_controlvanne_admin_carte_maintenance.py` | **Nouveau** — 2 tests |

L'autocomplétion s'appuie sur les `search_fields` de `CarteCashlessAdmin`
(`tag_id`, `number`, `user__email`) : on tape le Tag ID **ou** le numéro imprimé.

Bonus sécurité : la vue d'autocomplétion de Django passe par
`CarteCashlessAdmin.get_queryset()`, déjà filtré sur le tenant courant.

Aucune migration.

## Vérification

- **TDD** : les 2 tests écrits **échouaient** avant le fix (widget raw_id, fuite
  cross-tenant), passent après.
- `manage.py check` : 0 issue.
- Suites controlvanne + admin cartes : **46 tests passent**.
- **Chrome (dev)** : flow complet rejoué. Le champ ouvre une liste des Tag ID du
  lieu, la frappe `33BC1D` filtre en direct, l'enregistrement aboutit
  (« Carte de maintenance « 33BC1DAA » ajoutée avec succès »).

## Périmètre vérifié

`RfidSession` porte aussi une FK vers `CarteCashless`, mais son admin est en
lecture seule (`has_add_permission` et `has_change_permission` → `False`) : pas
de formulaire, donc pas de pk forgeable. `CarteMaintenance` était le **seul**
admin controlvanne éditable exposant cette FK.
