# Migration Stripe Connect `_from_config` — 2026-05-16

**Date :** 2026-05-16
**Statut :** ✅ Terminé, en production sur branche `main-wizard`.
**Spec ouverte du chantier large** : [`01-stripe-migration-spec.md`](./01-stripe-migration-spec.md).

Ce document est un **récap factuel** de la migration concrète des 2 méthodes
Stripe Connect `_from_config` depuis `BaseBillet/views.py::Tenant` vers
l'app dédiée `PaiementStripe/`. Il a été réalisé en parallèle du cleanup
du flow legacy `/tenant/new/` (cf. `../ONBOARD/03-session-recap.md`
section 9).

---

## 1. Contexte

L'app `BaseBillet/views.py::Tenant` était un ViewSet DRF qui mélangeait
2 responsabilités disjointes :

1. **Création de tenant** via le flow legacy `/tenant/new/` — supprimé
   (remplacé par l'app `onboard/`).
2. **Configuration Stripe Connect d'un tenant existant**, déclenchée
   depuis l'admin Unfold (Configuration → Products → bouton "Créer et
   lier son compte Stripe" dans `CheckStripeComponent`). À conserver.

Pour ne pas garder la classe `Tenant` rien que pour ces 2 méthodes
Stripe, on les a migrées dans l'app `PaiementStripe/` (dédiée aux
paiements Stripe).

## 3. Comportement (inchangé fonctionnellement)

### 3.1 Initier l'onboarding

Trigger : l'admin tenant ouvre l'admin Unfold pour créer un Product
payant. Le composant `CheckStripeComponent`
(`Administration/admin/products.py:794-804`) lit
`Configuration.stripe_payouts_enabled`. Si False :
- Render `admin/product/checkstripe_component.html` avec le bouton
  `<a href="/stripe/onboard/from_config/">`.
- Clic → `StripeConnectOnboardingViewSet.onboard_from_config()` :
  1. Lit `Configuration.get_solo()`, récupère `id_acc_connect` (ou en
     crée un nouveau via `Configuration.get_stripe_connect_account()`).
  2. Forge une `stripe.AccountLink.create(type="account_onboarding", ...)`.
  3. Redirige vers l'URL hosted Stripe (compte d'onboarding sécurisé).

### 3.2 Retour de Stripe

Stripe redirige vers `/stripe/onboard/return_from_config/<id_acc_connect>/`
après que l'admin a saisi ses infos (IBAN, identité, KYC…).
`onboard_return_from_config()` :
1. Vérifie `info_stripe.details_submitted` via API Stripe.
2. Met à jour `Configuration.stripe_payouts_enabled` si
   `info_stripe.payouts_enabled == True`.
3. Affiche le template `paiementstripe/after_onboard_stripe.html` avec
   un message succès ou pending.

### 3.3 Gestion d'erreurs

- `stripe.AccountLink.create` raise `InvalidRequestError` (compte Stripe
  invalide / expiré) → on vide `Configuration.stripe_connect_account` et
  on relance (recréation propre via `get_stripe_connect_account()`).
- `stripe.Account.retrieve` raise → 404 (Http404).
- Le `return_url` et `refresh_url` pointent sur le même endpoint
  (`return_from_config/<id>/`) : si Stripe a besoin de rafraîchir (lien
  expiré 1h), il revient au même endroit.

## 4. Tests manuels recommandés (avant de fermer la session)

### 4.1 Cas nominal — tenant SANS Stripe configuré

1. Connecte-toi à l'admin Unfold d'un tenant `WAITING_CONFIG` sans
   Stripe.
2. Va dans Products → Add product. Le `CheckStripeComponent` doit
   afficher le bouton "Créer et lier son compte stripe".
3. Clic. Tu dois être redirigé vers Stripe (URL `https://connect.stripe.com/...`).
4. Remplis le formulaire Stripe (mode test : IBAN factice
   `FR1420041010050500013M02606`, identité Jean Dupont, etc.).
5. Stripe te renvoie vers `/stripe/onboard/return_from_config/<id>/`.
6. Tu dois voir le template `after_onboard_stripe.html` avec le message
   "Your stripe account has been registered. Thank you very much!".
7. Refresh la page Products → le bouton "Créer et lier" doit avoir
   disparu (remplacé par "Votre compte stripe est connecté !").
8. En DB : `Configuration.stripe_payouts_enabled == True`.

### 4.2 Cas erreur — Stripe ne valide pas l'onboarding

1. Same setup. Clic sur le bouton.
2. Sur Stripe, abandonne sans remplir tout (clique "Save and finish later").
3. Stripe te renvoie sur `return_from_config/<id>/`.
4. Tu dois voir le message "You do not seem to have completed your
   Stripe registration. Please contact us if you need help."
5. En DB : `Configuration.stripe_payouts_enabled == False` (inchangé).

### 4.3 Cas exotique — Compte Stripe supprimé côté Stripe

1. En console Stripe Dashboard, supprime manuellement le compte
   connecté `id_acc_connect` du tenant.
2. En base, garde `Configuration.stripe_connect_account = "acct_..."`
   (l'ID supprimé).
3. Clic sur "Créer et lier" → `onboard_from_config` raise
   `InvalidRequestError` lors de `AccountLink.create`.
4. Le code catch, vide `stripe_connect_account = None`, en recrée un
   nouveau via `get_stripe_connect_account()`, et relance
   `AccountLink.create` avec ce nouveau ID.
5. Redirige vers Stripe normalement.

## 5. Points d'attention

### 5.1 Le `return_url` Stripe ne peut pas être un path relatif

L'`AccountLink.create()` exige une URL absolue HTTPS. On utilise
`tenant.get_primary_domain().domain` pour la construire. **Risque** : si
le primary_domain n'est pas configuré (cas pathologique), `.domain`
crashera. Pas de fallback dans le code actuel. Acceptable car un tenant
créé via la chaîne `wc.create_tenant()` a TOUJOURS un primary_domain
(cf. `BaseBillet/validators.py:969-973`).

### 5.2 Le `Tenant` ViewSet n'existe plus

Si du code externe importait `BaseBillet.views.Tenant`, il crashera à
l'import. `grep -rn "from BaseBillet.views import Tenant"` confirme :
aucun import résiduel.

### 5.3 Le template `after_onboard_stripe.html` a été DÉPLACÉ

Tout code Python qui référençait
`reunion/views/tenant/after_onboard_stripe.html` doit pointer sur
`paiementstripe/after_onboard_stripe.html`. `grep -rn` confirme : seul
notre nouveau code (PaiementStripe/views.py) le référence.

### 5.4 Stripe test mode

`RootConfiguration.get_stripe_api()` choisit la clé en fonction de
`stripe_mode_test`. Les tests doivent utiliser le mode test.
Configuration mode test : `RootConfiguration.stripe_mode_test = True` +
clés `stripe_test_api_key` / `stripe_test_publishable_key` configurées.

## 6. Tests automatisés

**Aucun test pytest n'a été ajouté pour cette migration.** Raison : les
2 méthodes appellent l'API Stripe directement (`stripe.AccountLink.create`,
`stripe.Account.retrieve`). Tester sans `stripe-mock` ou un mock lourd
ne ferait que tester le mock. Les tests manuels (cf. section 4) sont
suffisants pour ce périmètre.

**Si on veut industrialiser plus tard** :
- Mock `stripe.AccountLink.create` retournant `{"url": "https://stripe.com/...""}`
  → assert le redirect.
- Mock `stripe.Account.retrieve` retournant `{"details_submitted": True,
  "payouts_enabled": True}` → assert `Configuration.stripe_payouts_enabled
  == True`.
- Cas erreur : mock raise `InvalidRequestError` → assert flow de fallback.

## 7. Reprise en cas de bug

### 7.1 Bouton "Créer et lier son compte Stripe" ne s'affiche pas

- Vérifier que `Configuration.stripe_payouts_enabled == False` côté
  tenant (sinon le composant affiche un message OK et pas de bouton).
- Vérifier que le `CheckStripeComponent` est bien rendu dans le form
  Product de l'admin (`@register_component` dans
  `Administration/admin/products.py:793`).

### 7.2 Clic sur le bouton renvoie 404 ou 500

- Vérifier que `TiBillet/urls_tenants.py` inclut bien
  `path('stripe/', include('PaiementStripe.urls'))`.
- Vérifier que `PaiementStripe/urls.py` est peuplé (pas vide / commenté).
- Tester l'URL directement : `https://<tenant>/stripe/onboard/from_config/`
  doit renvoyer un 302 vers Stripe.

### 7.3 Retour de Stripe sur la mauvaise URL

- Le `return_url` est forgé dans `onboard_from_config()` à partir de
  `tenant.get_primary_domain().domain`. Si le domain primary du tenant
  est mal configuré (`is_primary=False` partout), `.domain` crashera.
- Solution : `python manage.py shell` puis
  `Domain.objects.filter(tenant__schema_name='<schema>')` + vérifier
  `is_primary=True` sur un seul.

### 7.4 `Configuration.stripe_payouts_enabled` ne passe pas à True

- Vérifier que `info_stripe.payouts_enabled` retourne True côté Stripe
  Dashboard. Stripe peut mettre du temps à activer un compte (KYC
  manuel parfois).
- L'admin peut forcer à True manuellement via le shell si Stripe
  confirme côté Dashboard (`Configuration.objects.first(); c.stripe_payouts_enabled = True; c.save()`).
